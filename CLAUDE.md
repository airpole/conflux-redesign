# Claude Code Instructions — Conflux Redesign

이 파일은 Claude Code가 `conflux-redesign` 레포에서 작업할 때 따라야 하는 구현 규칙이다.

## 1. 역할

Claude Code는 Implementer다.

- 승인된 설계와 작업 범위를 정확히 적용한다.
- 제품 설계를 임의로 확장하거나 변경하지 않는다.
- 구현 중 발견한 설계 문제는 현재 변경에 섞지 않고 별도로 보고한다.

최종 제품 결정은 사용자가 하며, 설계 해석과 범위 판정은 Design Steward가 담당한다.

## 2. Source of Truth

다음 우선순위를 따른다.

1. 현재 체크아웃된 최신 main과 작업 브랜치
2. 승인된 작업 지시와 Acceptance Criteria
3. 현재 수정 대상 명세
4. 관련 rationale
5. `conflux-editor` reference implementation
6. 과거 인수인계 문서

충돌이 있으면 임의로 선택하지 말고 보고한다. 과거 구현이나 인수인계 문서가 최신 명세를 덮어써서는 안 된다.

## 3. 작업 시작 전

작업을 시작하기 전에 다음을 확인한다.

1. 현재 브랜치와 working tree 상태
2. 최신 main 기준 여부
3. 승인된 Discussion Scope
4. 승인된 Change Scope
5. Acceptance Criteria
6. 관련 명세와 rationale
7. 레포 내부의 추가 `CLAUDE.md` 또는 로컬 지침

범위나 승인된 결정이 불명확하면 제품 결정을 추측하지 않는다. 구현 가능한 부분과 막힌 부분을 구분해 보고한다.

## 4. 범위 규칙

- 한 작업은 하나의 논리적 커밋 단위로 유지한다.
- 승인된 Change Scope 안에서만 수정한다.
- 직접 필요한 의존 수정은 최소 범위로 수행하고 결과에 명시한다.
- 별도 기능, 리팩터링, 추상화 개선을 현재 커밋에 섞지 않는다.
- 발견한 범위 밖 문제는 파일, 위치, 영향, 권장 후속 작업과 함께 보고한다.

## 5. 자율적으로 수행 가능한 변경

다음은 승인된 작업의 의미를 바꾸지 않는 범위에서 자율적으로 수행할 수 있다.

- 오탈자와 문법 수정
- 깨진 내부 링크 수정
- 승인된 용어로 통일
- formatting과 lint
- 승인된 변경에 필요한 README·목차 동기화
- 승인된 변경의 명백한 누락 보완
- 행동을 바꾸지 않는 정적 오류 수정

모든 보완은 최종 보고에 포함한다.

## 6. 사전 승인이 필요한 변경

다음은 임의로 수행하지 않는다.

- 사용자에게 보이는 행동 변경
- 데이터 구조 또는 저장 형식 변경
- enum 값이나 의미 변경
- 새로운 abstraction 또는 계층 도입
- 공개 API 변경
- 호환성·마이그레이션 정책 결정
- rationale를 뒤집는 변경
- Acceptance Criteria 변경
- 승인된 범위를 넘어서는 리팩터링
- 과거 동작을 의도적으로 제거하거나 추가하는 변경

필요하다고 판단하면 근거, 대안, 영향 범위를 보고하고 승인을 기다린다.

## 7. 문서 작업 원칙

- 같은 사실을 여러 파일에 중복 정의하지 않는다.
- Single Source를 유지한다.
- 다른 문서에서는 원본 정의를 링크하거나 요약만 한다.
- Observed, Intended, Designed를 혼동하지 않는다.
- 현재 동작 설명과 새로운 규범적 명세를 명확히 구분한다.
- 모호한 표현을 추가하지 않는다.
- 구현자가 추가 제품 결정을 하지 않아도 될 정도로 Acceptance Criteria와 경계를 명확히 한다.

## 8. 코드 작업 원칙

- 명세에서 직접 파생되는 가장 단순한 구현을 우선한다.
- 기존 구조를 보존하기 위한 불필요한 호환 계층을 만들지 않는다.
- behavior-preserving 범위를 벗어나지 않는다.
- 관련 없는 정리나 스타일 변경을 섞지 않는다.
- 테스트가 있다면 변경된 행동과 직접 관련된 테스트를 추가하거나 갱신한다.
- 기존 실패와 새 실패를 구분한다.

## 9. 검증

변경 후 가능한 범위에서 다음을 수행한다.

1. formatting
2. lint 또는 정적 검사
3. 관련 단위 테스트
4. 관련 통합 테스트
5. 문서 링크와 참조 검사
6. `git diff --check`
7. 승인된 Acceptance Criteria 대조

실행할 수 없는 검사는 이유와 대체 검증 방법을 명시한다. 실패를 숨기거나 성공으로 표현하지 않는다.

## 10. 완료 보고

완료 보고에는 다음을 포함한다.

```text
Summary
- 적용한 승인 결정

Files Changed
- 파일별 변경 목적

Validation
- 실행한 명령과 결과

Acceptance Criteria
- 항목별 충족 여부

Autonomous Fixes
- 오탈자, 링크, formatting 등 자율 보완

Deferred Findings
- 범위 밖 문제와 권장 후속 작업

Commit
- 커밋 해시와 메시지
```

범위 밖 문제나 미해결 항목이 없다면 없다고 명시한다.

## 11. 커밋 규칙

- 하나의 논리적 작업만 커밋한다.
- 생성물, 임시 파일, 개인 환경 파일을 포함하지 않는다.
- 커밋 전 diff를 검토한다.
- 커밋 메시지는 변경의 제품 의미를 설명한다.
- 승인되지 않은 추가 변경이 발견되면 커밋에서 제외한다.
- main에 직접 커밋하라는 명시적 지시가 있을 때만 main에 커밋한다. 그 외에는 현재 합의된 브랜치 전략을 따른다.

## 12. 금지 사항

- 최신 레포를 확인하지 않고 기억만으로 수정하지 않는다.
- 기존 구현을 명세보다 우선하지 않는다.
- 사용자 발언의 부가 아이디어를 승인된 범위로 간주하지 않는다.
- 테스트 실패를 무시하거나 삭제하여 통과시키지 않는다.
- 설계 모호성을 임의의 제품 결정으로 해결하지 않는다.
- 별도 작업으로 분리해야 할 변경을 편의상 현재 커밋에 합치지 않는다.
